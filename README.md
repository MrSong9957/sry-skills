# sry-skills

个人收集的技能（Skill）仓库，包含多种工程自动化技能，可用于代码简化、错误修复、前端设计、UI/UX优化等多种场景。

## 项目介绍

sry-skills是一个基于个人收集的技能（Skill）仓库，包含了多种用于工程自动化的技能定义和实现。每个技能专注于特定领域的任务，可独立使用或组合使用，帮助用户快速完成各种工程任务。

## 核心特点

- **技能丰富**：包含20+种不同领域的技能，涵盖代码、设计、测试、项目管理等多个方面
- **独立封装**：每个技能独立封装，可单独使用
- **易于扩展**：采用模块化设计，便于添加新技能
- **详细文档**：每个技能都有完整的使用说明和示例
- **多种应用场景**：适用于开发、设计、测试、调试等多种工程场景

## 技能列表

<a id="brainstorming"></a>
### brainstorming
- **名称**：头脑风暴
- **描述**：用于创意生成和问题解决的头脑风暴技能
- **使用场景**：
  - 项目初期的创意生成
  - 解决复杂问题
  - 探索新功能的可能性
  - 团队协作讨论

<a id="code-simplifier"></a>
### code-simplifier
- **名称**：代码简化器
- **描述**：在保留全部功能的前提下，简化并优化代码，提升其清晰度、一致性与可维护性
- **使用场景**：
  - 简化复杂代码结构
  - 优化代码可读性
  - 统一代码风格
  - 移除冗余代码

<a id="dispatching-parallel-agents"></a>
### dispatching-parallel-agents
- **名称**：并行代理调度
- **描述**：用于处理两个或更多独立任务的并行代理调度技能
- **使用场景**：
  - 同时处理多个独立任务
  - 优化任务执行效率
  - 处理无共享状态的并行任务

<a id="error-fix"></a>
### error-fix
- **名称**：错误修复专家
- **描述**：根据用户输入的错误信息，进行系统分析、排查问题原因、制定并执行修复方案，最终完成错误修复任务
- **使用场景**：
  - 开发人员排查代码错误
  - 系统管理员分析系统故障
  - 技术支持人员快速定位问题
  - 普通用户解决软件问题

<a id="error-notebook"></a>
### error-notebook
- **名称**：错误笔记本
- **描述**：浏览对话历史，识别错误并以日记式追加方式记录到 docs/error-notebook.md
- **使用场景**：
  - 记录开发过程中的错误
  - 建立错误知识库
  - 分析错误模式
  - 为团队提供错误参考

<a id="executing-plans"></a>
### executing-plans
- **名称**：执行计划
- **描述**：在有审核检查点的单独会议中执行包含独立任务的实施计划
- **使用场景**：
  - 执行复杂的多步骤计划
  - 项目实施
  - 任务管理

<a id="finishing-a-development-branch"></a>
### finishing-a-development-branch
- **名称**：完成开发分支
- **描述**：通过提供结构化的合并、拉取请求或清理选项，指导完成开发工作
- **使用场景**：
  - 完成功能开发后处理分支
  - 准备代码合并
  - 清理开发环境

<a id="frontend-design"></a>
### frontend-design
- **名称**：前端设计专家
- **描述**：创作设计精良、达到生产级标准的独特前端界面
- **使用场景**：
  - 构建Web组件
  - 设计网页或应用界面
  - 创建HTML/CSS布局
  - 美化Web UI

<a id="json-canvas"></a>
### json-canvas
- **名称**：JSON Canvas编辑器
- **描述**：创建并编辑包含节点、连线、分组及关联关系的JSON画布文件（.canvas格式）
- **使用场景**：
  - 制作思维导图
  - 创建流程图
  - 设计可视化画布
  - 在Obsidian中使用画布文件

<a id="receiving-code-review"></a>
### receiving-code-review
- **名称**：接收代码审查
- **描述**：在收到代码审查反馈后、实施建议之前使用，尤其是当反馈看起来不清晰或在技术上存在疑问时
- **使用场景**：
  - 处理代码审查反馈
  - 实施代码改进建议
  - 确保代码质量

<a id="requesting-code-review"></a>
### requesting-code-review
- **名称**：请求代码审查
- **描述**：在完成任务、实施主要功能或在合并之前使用，以验证工作是否符合要求
- **使用场景**：
  - 请求代码审查
  - 确保代码质量
  - 获得团队反馈

<a id="subagent-driven-development"></a>
### subagent-driven-development
- **名称**：子代理驱动开发
- **描述**：在当前会话中执行包含独立任务的实施计划时使用
- **使用场景**：
  - 复杂项目的开发
  - 多团队协作
  - 分阶段任务执行

<a id="superpowers"></a>
### superpowers
- **名称**：超能力系统
- **描述**：一个运行在严格项目结构下的工程自动化系统，负责解析用户意图、匹配技能、执行多阶段工作流
- **使用场景**：
  - 复杂工程任务的自动化
  - 多技能协同工作
  - 全流程工程自动化

<a id="systematic-debugging"></a>
### systematic-debugging
- **名称**：系统性调试
- **描述**：遇到任何程序缺陷、测试失败或意外行为时，在提出修复方案之前使用
- **使用场景**：
  - 复杂bug的调试
  - 测试失败的分析
  - 系统故障的排查

<a id="test-driven-development"></a>
### test-driven-development
- **名称**：测试驱动开发
- **描述**：在编写实现代码之前，实施任何功能或错误修复时使用
- **使用场景**：
  - 测试驱动的功能开发
  - 错误修复的测试验证

## 技能定义格式

每个技能都包含以下核心部分：

1. **名称（name）**：技能的唯一标识符
2. **描述（description）**：技能的功能和用途
3. **使用场景**：技能适用的具体场景
4. **输入（inputs）**：技能所需的输入参数
5. **输出（outputs）**：技能执行后的输出结果
6. **执行步骤（steps）**：技能的具体执行流程
7. **前置条件（preconditions）**：执行技能前必须满足的条件

## 使用方法

1. 选择需要的技能
2. 查看该技能的SKILL.md文件，了解详细使用方法
3. 根据技能要求提供输入
4. 执行技能并获取输出
